## PHASE 1 — Local Package Setup

### Step 1: Set your identity in DESCRIPTION

Open `DESCRIPTION` and replace the placeholder fields:

```
Authors@R: c(
    person("Your",  "Name",
           email   = "your.real@email.com",
           role    = c("aut", "cre"),
           comment = c(ORCID = "0000-0000-0000-0000")))
```

> **CRAN rule:** The `cre` (maintainer) email must be a real, monitored
> address. CRAN will email you when the package needs attention.

---

### Step 2: Verify the folder structure

Your package folder must look exactly like this:

```
rDataRecon/
├── DESCRIPTION                        # package metadata
├── NAMESPACE                          # exports (auto-generated by roxygen2)
├── LICENSE                            # MIT year + copyright holder
```

## PHASE 2 — R CMD CHECK (The Gate to CRAN)

### Step 6: Run R CMD CHECK locally

```r
devtools::check()
```

**You must achieve:**

```
-- R CMD check results -----------------------------------------------
Duration: 45s

0 errors v | 0 warnings v | 0 notes v
```

### Common CHECK failures and fixes

| Error / Warning | Cause | Fix |
|---|---|---|
| `no visible binding for global variable` | Bare variable name in code | Add `utils::globalVariables()` |
| `checking DESCRIPTION meta-information ... WARNING` | Missing or malformed fields | Fix `Authors@R`, `License`, `URL` |
| `Undocumented arguments in documentation object` | `@param` missing for a function arg | Add `@param` tag for every argument |
| `Examples with CPU time > 5s` | Example too slow | Wrap in `\donttest{}` |
| `package on CRAN, version already present` | Version already exists | Bump `Version` in `DESCRIPTION` |
| `checking for non-standard things in the check directory` | Tests leave temp files | Use `withr::local_tempdir()` in tests |

## PHASE 4 — Final Pre-Submission Checklist

```r
# 1. Re-document
devtools::document()

# 2. Full test suite — must be 0 failures
devtools::test()

# 3. Local R CMD CHECK — must be 0 errors, 0 warnings
devtools::check()

# 4. Build the source tarball
devtools::build()
# Creates: ../rDataRecon_1.0.0.tar.gz

# 5. Check the built tarball (most accurate simulation of CRAN check)
rcmdcheck::rcmdcheck("../rDataRecon_1.0.0.tar.gz",
                     args     = c("--as-cran"),
                     error_on = "warning")

# 6. Win-builder checks
devtools::check_win_devel()
devtools::check_win_release()

# 7. Spell-check documentation
devtools::spell_check()

# 8. Check URLs are reachable
install.packages("urlchecker")
urlchecker::url_check()
```

### Step 13: Submit to CRAN

```r
devtools::release()
```

This command:
1. Runs a final interactive checklist (~20 questions)
2. Builds the source tarball `rDataRecon_1.0.0.tar.gz`
3. Opens the CRAN submission form at https://cran.r-project.org/submit.html

**Or submit manually:**

1. Go to: https://cran.r-project.org/submit.html
2. Upload: `rDataRecon_1.0.0.tar.gz`
3. Fill in your name and email (must match `cre` in DESCRIPTION)
4. Paste `cran-comments.md` content in the "Optional comments" box
5. Click Submit

### Step 14: Click the confirmation email

Within minutes, CRAN sends a confirmation email with a link.
**You must click it** to finalise the submission.

## PHASE 6 — After Submission

### Timeline

| Day | What happens |
|-----|-------------|
| 0   | You submit + click confirmation email |
| 0–1 | Automated CRAN incoming checks run |
| 1–7 | Human CRAN volunteer reviews the package |
| 1–7 | You receive ACCEPTED or a list of issues to fix |
| 7–14 | Package appears on CRAN (after acceptance + propagation) |

### If CRAN requests changes

| CRAN note | What to do |
|---|---|
| "Please add `\value` to all exported functions" | Add `@return` to every `@export` function |
| "Please use `on.exit()` to clean up side effects" | Wrap file writes in `on.exit(unlink(tmp))` |
| "Examples must not write to the user's home directory" | Use `tempfile()` in all examples that write files |
| "Please omit `LazyData: true` — package has no data" | Remove `LazyData` from DESCRIPTION |
| "Executable files not allowed" | Run `chmod -x` on any scripts |

Fix, bump micro version (`1.0.0` → `1.0.1`), re-run all checks, resubmit.

## Key CRAN Policies

1. **No internet access in examples or tests** — use `skip_if_offline()`.
2. **No writing to home directory** — all file output must use `tempfile()`.
3. **`\dontrun{}` vs `\donttest{}`** — use `\donttest{}` for slow examples;
   `\dontrun{}` only for genuinely non-runnable examples.
4. **Check time limit** — total check time must be under 10 minutes.
5. **Re-submission wait** — wait at least 2 weeks between CRAN submissions
   unless fixing a critical bug.
6. **Deprecated dependencies** — if a dependency is archived on CRAN,
   your package will also be archived. Keep dependencies minimal.
